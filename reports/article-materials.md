## 附：环境版本演进时间线

文章的核心叙事虽聚焦于交互模式，但**底层平台的变更**同样深刻影响了每一个时期的用户感知。

| 时间 | OpenClaw 版本 | 主力模型 | 备注 |
|------|--------------|---------|------|
| 3月24日 | v3.4 | Qwen | 原始 handler 上线，Qwen 首次接触积分记账场景 |
| ~4月初 | v3.4 | 切换至 KM 2.6 | 模型更换导致解析风格变化？日志中未见突变 |
| ~4月中旬 | 升级（约+2周） | KM 2.6 | OpenClaw 从 v3.4 升级，agent 行为可能改变 |
| ~4月底~5月初 | 再次升级 | 切换至 MiniMax 2.7 | 模型+agent 同时更新，P4→P5 转折点与此高度吻合 |
| ~5月10日 | ~v5.10 | MiniMax 2.7 | 最新版本，200K 上下文 |

### 对分析的影响

1. **模型切换是隐形变量**——Qwen→KM2.6、KM2.6→MiniMax2.7 的切换都没有明确日志记录，但 LLM 的解析风格一定会随模型改变。P1 的"+0分综合症"和 P3 的"负号歧义"未必是同一个 handler 代码导致的。
2. **OpenClaw 版本升级改变 Agent 行为**——OpenClaw 的版本升级可能改变消息路由、上下文长度、甚至 prompt 拼接逻辑。
3. **重构项目应记录这两项数据**——每次记账都记录当时的 `model_name` 和 `agent_version`，做到"每笔账都知道是谁算的"。
4. **重构项目代码中已在 pipeline.py 用环境变量指定模型**：`KIDS_POINTS_LLM_MODEL`，但未持久化到数据库。

## 一、五个时期的精细划分

从 input.log 的时间线和交互特征来看，用户与 Agent 的互动不是简单的"进步"，而是一条**来回震荡的进化曲线**。建议将原三个时期扩展为**五个时期**：

| 时期 | 时间 | 时长 | 输入特点 | 核心矛盾 | 问题数量等级 |
|------|------|------|----------|----------|------------|
| **P1 盲测期** | 3月24日 → 3月28日 | 5天 | 命令式试探，像在调测API | 用户不知道AI的边界在哪 | ⭐⭐⭐ |
| **P2 乱序期** | 3月29日 → 4月8日 | 11天 | 表达多样化，消费和加分混在一起 | 同一个意图五花八门的说法 | ⭐⭐⭐⭐⭐ |
| **P3 范式期** | 4月9日 → 4月14日 | 6天 | 用户发明固定格式，规则化输入 | LLM的不稳定性暴露（-1歧义、调账首次出现） | ⭐⭐⭐⭐ |
| **P4 秩序与崩坏期** | 4月15日 → 4月30日 | 16天 | 进入日常使用，信任建立 | 信任带来滥用/测试/重复记账积累 | ⭐⭐⭐⭐⭐ |
| **P5 默契与遗忘期** | 5月1日 → 5月17日 | 17天 | 极简表达与自然语言并行 | 用户忘记规则、调账连锁错误 | ⭐⭐⭐ |

## 二、P1：盲测期（3月24日 - 3月28日）

**核心特征：双方都在探测对方的语言系统。用户在"黑盒测试"Agent 的边界。**

### 时序脉络

**Day 1（3月24日）：开场——像调接口**
```
"学习积分 测试 +1 分"              ← 第一条，格式像API参数
"学习积分 测试减 -1 分"            ← 3分钟后，测试扣分
"学习积分语文作业 1 分"            ← 省去空格，"学习积分"当命令前缀
"不是消费" "学习积分消费 0分"       ← 对"消费"概念的理解在形成
"消费积分0分？" "消费积分 1分"      ← 用户对"消费/积分"两个词的位置还在试
"积分消费 2 分"                    ← "积分消费"这个顺序最后由本例稳定下来
```

**Day 1 关键：用户在 2 小时内发了约 15-20 条试探消息。** 包括各种排列组合：
- 谁在前？（"学习积分消费" vs "积分消费" vs "消费积分"）
- 单位？（"1分" vs "1" vs "1个积分"）
- 分隔符？（空格、逗号、冒号、无分隔符）

**Day 2（3月25日）：补录机制被发明**
```
"补录 2026-03-25 积分：语文作业 +1 分，数学作业 +1 分，英语作业 +1 分，口算（没有全对）+1 分，家里写字作业 +1 分"
```
→ 用户发现无法直接说"昨天的"，于是发明了"补录 YYYY-MM-DD"前缀。这是**用户主动适配系统的第一个证据**。

**Day 3-5（3月26日-3月28日）：重复输入成为逃生策略**
```
"今天完成了语文作业 +1 分，完成了英语作业 +1 分"  → +0分（提取成"自主申报"）
"今天完成了语文作业 +1 分，完成了英语作业 +1 分"  → +2分（同一秒重发，反而正确！）
```

**第3天上演的"手写打卡"：**
```
"写字学习记分：写字打卡完成了第15天 +1 分"
"写字学习记分：手写 2 课手写 +1 分"
"口算学习记分：口算 2 课口算 全对 0 分"     ← "全对 0 分"？!
"口算学习记分：口算 2 课口算 "               ← 用户被逼出了一个新的格式前缀
"今天完成了 书写 书写打卡 +1 分 口的算 2篇全对的"
```
→ 用户尝试发明了新的命令前缀模式，但 Agent 对"全对却得了0分"这类语义矛盾完全没有识别。

**Day 5（3月28日）：第一条纯字符串被理解错**
```
"英赛尔学习：记2分"          ← 安全模式：+0分（Agent不认识"英赛尔"）
"英赛尔学习：2"             ← 第二天尝试：+0分
"英赛尔学习：1"             ← 还没放弃：+0分
```

### P1 暴露的问题

| 问题 | 表现 | 根源 |
|------|------|------|
| **命令前缀不统一** | 用户试了"学习积分"、"积分消费"、"消费积分"、"学习记分"等十几种前缀 | 用户不知道系统期待什么格式 |
| **Agent 理解不稳定** | 同一条消息重发一次结果不同 | LLM 解析的非确定性 |
| **找不到"不知道"** | 英赛尔被+0分，而不是被拒绝 | Agent 不会说"我不认识这个" |
| **格式试探成本高** | 用户发了十几条试出可用格式 | 无引导/无样板提示 |
| **+0 分综合症** | "不确定就给0分"成为系统的安全模式 | 不安全——给0分比直接拒绝更隐蔽地污染数据 |

## 三、P2：乱序期（3月29日 - 4月8日）

**核心特征：用户不再测试格式了，开始"用"了。但表达能力太多样，Agent 跟不上。**

### 时序脉络

**3月29日-4月1日：消费表达的爆发（4种不同说法在4天内涌现）**
```
"积分消费 2 分 - 早上把卷子发卷烂了"     ← 格式1："积分消费 X 分 - 理由"
"积分消费 买铅笔花了 5 分 - 丢了顾安的铅笔" ← 格式2：扩展理由
"买了个毽子，要消费了 6 分"               ← 格式3：先说买了什么
"积分消费 买小风筝花了 15 分"             ← 格式4：组合
```

**4月3日：结构化列表出现——但用户自己也不确定写法**
```
"记一下昨天的积分吧：数学错题改正得A+加2分，语文生字错题改正加2分，帮妈妈做家务（晒衣服、扫地…）各加2分，数学口算全对加2分"
```
→ 用自然语言列了4项，让 Agent 自己算总分。这是**信任开始建立的标志**——用户不再写"消费8分"而写"买了什么、花了多少"。

**4月4日-7日：两条跨天消息的"差异陷阱"**
```
"昨天语文生字改正加2分，数学错题改正加2分，口算全对加2分"  → 算对
"昨天语文生字改正加2分，数学错题改正加2分，口算全对加1分"  → 只改了1个数字，但同一模式
```
→ 用户把前天的事情也集中到今天说，但**同一个事件模式在日志中出现多次，无法区分是重复还是新记录**。

**4月7日：家庭方言首次未被识别**
```
"火花小数学加1分"  → 解释为"数学作业"（脑补）
```
→ Agent 把"火花小数学"（一个数学思维App）归类为"数学作业"。在没有纠正机制的情况下，这种脑补会永久污染类型统计。

### P2 暴露的问题

| 问题 | 表现 | 严重程度 |
|------|------|----------|
| **消费表达歧义爆炸** | 7天内出现4种不同句式表达"扣分" | ⭐⭐⭐⭐⭐ |
| **多项目合并的边界** | "记一下昨天的积分"包含4-6项，Agent漏项或重复 | ⭐⭐⭐⭐ |
| **家庭方言盲区** | "火花小数学"被脑补为"数学作业" | ⭐⭐⭐ |
| **跨天消息无法追踪** | 同一天说昨天的事，昨天的日志已经结算了 | ⭐⭐⭐⭐ |
| **无校验的自然语言** | "全对+0分"这种矛盾通过 | ⭐⭐⭐⭐⭐ |

## 四、P3：范式期（4月9日 - 4月14日）

**核心特征：用户开始系统性地使用"固定格式"来驯服 Agent 的不确定性。但 LLM 的固有缺陷显露出来。**

### 时序脉络

**4月9日-11日：结构化列表的巅峰——"不要省略"风格**
```
"1. 口算加2分 2. 语文写字加1分 3. 英语作业加1分 4. 读书笔记加1分 5. ABC Reading加2分 6. 练字加1分"
```
→ 编号列表 + "项目名" + "加X分"。这是人类面对不可靠 AI 时进化出的最优通信协议：**让 Agent 做最少的推理，让人类承担最多的结构。**

**4月12日：负号歧义第一次暴露**
```
"吃口香糖 -1 分"    → LLM解析成 +1分（把"-1"的横线当连字符）
"积分消费 吃口香糖花了 1 分"  → 用户立即纠正，这次是 -1分（正确）
```
→ **第一次明确的负号歧义暴露。** 同种输入(含"-"号)，LLM 在不同上下文中的解析结果不同。用户被迫"补一条保险"——用"积分消费"前缀强制扣分意图。

**4月13日：负号歧义再次出现 + 用户"补刀"导致重复**
```
"买口香糖 -1 分"    → 这次解析成 -1分（正确，但用户不信任）
"积分消费 买口香糖花了 1 分"  → 又记了一次 -1分（重复！）
"
最终余额：55.0分"（两条扣分前余额57，扣了两次55）→ 重复记账确认
```
→ 第二次负号歧义：**同样的"-1分"格式，这次 LLM 解析对了，但用户不放心又补了一条保险，结果重复记账。** 这是用户和 LLM 之间的"信任鸿沟"——用户已经不相信 LLM 会稳定处理"-"号了。

**4月13日：第一条调账（一次性成功）**
```
"调账：修正看动画片 1 集扣分（应扣 -5 分但扣了 -10 分，调账 +5 分）"
  → +5分（成功，余额37分）
```
→ 第一次调账反而很顺利。因为用户描述精确（说了应扣多少、实扣多少、补多少）。这说明**调账是否能成功，核心在于用户能不能说清楚"从哪到哪"**。

**4月13日-14日："英赛尔"词汇终于被理解**
```
"英赛尔学习 +1 分"  → +1分（正确！）
"英赛尔 +1 分"     → +1分
```
→ 从 P1 的"英赛尔=+0分"到 P3 的"英赛尔=+1分"，大约花了 20 天。家庭方言的"通过测试"周期长达三周。

### P3 暴露的问题

| 问题 | 表现 | 严重程度 |
|------|------|----------|
| **"-"号的非确定性解析** | 同格式 "-1分" 先后解析为加法和减法 | ⭐⭐⭐⭐⭐ |
| **用户"补刀"导致重复** | 对 LLM 的不信任→多发一条保险→重复记账 | ⭐⭐⭐⭐⭐ |
| **家庭方言学习周期长** | "英赛尔"从被理解到稳定处理用了20天 | ⭐⭐⭐ |
| **调账的隐含语法** | 成功的调账依赖用户说清楚"应扣/实扣/差额" | ⭐⭐⭐⭐ |
| **LLM 的"惯性"** | 连续两天的"-X分"可能因 context 累积而产生不同结果 | ⭐⭐⭐ |

## 五、P4：秩序与崩坏期（4月15日 - 4月30日）

**核心特征：日常使用走向稳定，但信任积累导致的两个极端——深入使用暴露深坑 + "测试"导致系统崩坏。**

### 时序脉络

**4月15日：小数解析第一次崩盘（凌晨00:22）**
```
"看了半集动画片，扣 2.5 分"
  → 提取任务：看了半集动画片，扣 2. (-5分)    ← 小数"2.5"被截断为"2."
  → 积分变动：-5分
```
→ **LLM 把"2.5分"解析成了"2分"，但后面的"5"被当成了"5分"拼到描述里，最终结果变成"扣5分"。**

**4月15日 02:14：用户发现不对，调账——同样的坑再踩一次**
```
"调账加 2.5 分，因为昨天动画片只看了一半"
  → 提取任务：调账加 2.，因为昨天动画片只看了一半 (+5分)
  → 积分变动：+5分
```
→ **两次错误抵消了，但余额多了。而且完全没有人发现。** 这种静默错误是 LLM 作为记账员的致命弱点——错误不报，数据被污染，后面再也找不回真相。

**4月16日-20日：两种"复读"的谜之区别**
```
"拼音听写 +1 分"  "拼音听写 +1 分"    → 被当作两条不同记录（重复）
"改学校错题加2分" "改学校错题加2分"  → 也是两条（重复）
"帮家里干家务：扫地+1分，叠衣服+1分" → 正确解析为2条（但"扫地"和"叠衣服"换行写）
```
→ **原始系统的去重逻辑是字符串精确匹配？还是当天全量匹配？** 从日志看，"拼音听写 +1 分"完全相同的字符串被处理了两次——说明要么没去重，要么去重基于 message_id 但每一条飞书消息的 ID 不同。

**4月23日-27日：日常使用的巅峰——12项批量记账**
```
"1. 口算加2分 2. 语文写字加1分 3. ABC Reading加2分 4. 练字加1分
 5. 学校作业改错加2分 6. 英语作业加1分 7. 语文预习加2分
 8. 口算2加2分 9. 读书笔记加1分 10. 跳绳加1分
 11. 学习积分 晚睡扣1分 12. 学习积分 未定作业扣1分"
```
→ 12 条计在一组消息里，余额冲高到 133 分。用户完全进入了"日常使用"模式。

**4月27日：真正的灾难——同一消息被重复记账5次**
```
"在托管班完成学校课内作业加4分"  → +4分 余额117.4
"在托管班完成学校课内作业加4分"  → +4分 余额121.4
"在托管班完成学校课内作业加4分"  → +4分 余额125.4
"在托管班完成学校课内作业加4分"  → +4分 余额129.4
"在托管班完成学校课内作业加4分"  → +4分 余额133.4
```
→ **系统完全没有去重。** 余额在 5 秒内被污染 +16 分。而且这 5 条后面跟着用户的真实购买消费，整个余额序列从此不可信。

**4月28日："测试攻击"——系统完全失控**
```
"测试消费"                         → LLM脑补：买零食 (-30分)
"undefined"                        → 生成：undefined (NaN分)
"测试消费"                         → 又是买零食 (-30分)
"test"                             → 测试 (-10.5分)
"积分消费买零食花了30分"           → 买零食 (-30分)
"积分消费买零食花了30分"           → 买零食 (-30分) 又是
```
→ **7条消息、5分钟之内，余额从133分跌到-100.5分。** 最离谱的是 "undefined"——这不是人发的，极可能是飞书客户端或某个插件 bug 自动发送的。

**4月28日-30日：恢复与清理**
```
用户发现余额崩了，开始用"积分消费买XXX"做多笔消费正名 → 无法
```
→ 测试攻击造成的污染无法追溯，后续的日常使用在污染的余额上继续运行。

### P4 暴露的问题

| 问题 | 表现 | 严重程度 |
|------|------|----------|
| **小数解析永远不可靠** | "2.5"被LLM截断成"2.→-5分"，调账再踩一次 | ⭐⭐⭐⭐⭐ |
| **去重完全缺失** | 同一消息×5次，去重完全没有运作 | ⭐⭐⭐⭐⭐ |
| **对"测试"输入的零防御** | "test"→-10.5分、"undefined"→NaN，无任何拦截 | ⭐⭐⭐⭐⭐ |
| **余额污染后不可恢复** | 测试攻击+重复记账，余额永远对不上 | ⭐⭐⭐⭐⭐ |
| **静默错误** | 小数解析错但系统不报错，用户全靠运气发现 | ⭐⭐⭐⭐ |

## 六、P5：默契与遗忘期（5月1日 - 5月17日）

**核心特征：日常使用高度流畅，但深层问题（调账、重复、家庭方言扩容）在该时期埋的雷逐个引爆。**

### 时序脉络

**5月1日-7日：极简风格显现**
```
"晚睡扣2分"                              ← 没有"积分消费"前缀了
"方格训练加1分"                          ← 新的家庭词（舒尔特方格）
"晚睡扣4分（11点才睡）"                   ← 括号补充理由
"没记当日作业 -2"                         ← 用户用"-2"（负号直接写，信任高位）
"买冰激凌 -3"                            ← 直接"-3"格式，和3月的乱序期完全不一样
```
→ **用户回到自然语言了。** 与 P2 的"买了个毽子，要消费了6分"类似，但语气更自信——不再用"积分消费"前缀。因为他们已经知道，就算说"买冰激凌 -3"Agent 也能听懂。

**5月8日：为他人加分的复杂语义**
```
"帮王雯雯买冰棍，两根冰棍8分，给她加8分"
  → 第一次解析：买冰棍 (-8分)  ← 理解成"买=消费=扣分"
```
→ 系统把"给她加8分"中的"加8分"忽略了。这是一个语义级别的复杂场景：**同一条消息中同时出现"消费"和"加分"，且受益者是第三个人。**

**5月9日：调账灾难——来回拉锯4次**
```
→ 用户发现"帮王雯雯买冰棍"被扣了8分，开始调账

一次调账："调账：帮王雯雯买冰棍应为她加8分（收入），不是扣分"
  → 理解成：再扣16分（因为"为"+负面描述→被当成修正扣分）

二次调账："调账：帮王雯雯买冰棍给她加8分（收入）"
  → 又是扣16分

三次调账："调账：帮王雯雯买冰棍应为加8分（收入）"
  → 终于理解成+8分

四次修复："纠正调账错误：之前帮王雯雯买冰棍多扣了32分，补回24分"
```
→ **经典案例：一个简单的"加回8分"，花了4次调账才完成。** 整个过程产生了：
- 原记账：-8分（错误）
- 调账1：-16分（错上加错）
- 调账2：-16分（继续错）
- 调账3：+8分（终于对了）
- 调账4：+24分（修复前面3次错误）
- **净效果：正确应该是+8分，实际走了-8-16-16+8+24=-8分 → 等等算不清了**
- 事实是：谁也不知道最终余额对不对。

**5月16日：[object Object] 攻击——自动生成的垃圾消息**
```
"[object Object]"  ×8
```
→ 连续 8 条 "[object Object]"，全部被处理为"自主申报 +0分"。极可能是手机端飞书某个插件按钮生成的序列化错误。虽然+0分无害，但说明系统没有错误输入防御。

**5月17日：最终形态——高度结构化**
```
"1. 口算加1分 2. 语文写字加3分 3. ABC Reading加3分 4. 英赛尔加3分 5. 学校作业改错加2分"
```
→ 极致简洁：编号 + 项目名 + 分数，无多余词。跟 P4 的 12 项批量类似，但更干净。

**5月11日-17日：跨天复合消息**
```
"昨天语文写字加3分，ABC Reading没读，英赛尔加3分口算加一分"
  → 返回+7分（正确解析了"没读=+0分"）
```
→ "ABC Reading没读"——用户明确说某项目没有做，要求+0分。这说明用户已经接受了"做了加分、没做0分"的框架，并且在框架内能精确操作。

### P5 暴露的问题

| 问题 | 表现 | 严重程度 |

| **调账系统设计缺陷** | 调账是"直接加减余额"而非"先修正+自动补差额"，导致多次调账互相叠加 | ⭐⭐⭐⭐⭐ |
| **复杂语义（为他人）** | "帮X买，给她加分"中消费和加分混合 | ⭐⭐⭐⭐ |

## 八、关键论断与新内核

### 论断1：进化不是线性的，而是震荡的

五个时期揭示的不是"越来越好"的曲线，而是一个**周期性上升但不断震荡崩溃**的路径：

```
P1(盲测) → P2(乱序) → P3(范式) → P4(秩序+崩坏) → P5(默契+遗忘)
   ^          ^           ^           ^               ^
   |          |           |           |               |
  低效      混乱       驯服      稳定但脆弱        高效但不可靠
```

**在每个"稳定"期（P3、P5），系统都积累了未解决的深层问题，最终在下个时期爆发。**

### 论断2：三个最具代表性的人机互动 moment

**moment 1：用户说减、Agent 加（P3）**
"吃口香糖 -1 分"→+1分。这是最写实的一个 moment：用户以为自己在表达"减法"，AI 用语言模型的方式"理解"成了加法。双方没有对错，只是两个系统的"不兼容"。

**moment 2：同一条消息被记了5次账（P4）**
"在托管班完成学校课内作业加4分"→×5，余额+20分。去重不是"有了就行"，去重是记账系统的基石。

**moment 3：调账调了4次才说清楚（P5）**
一条"帮她加分"的请求，触发了4次调账、总额变动 ±120 分的混乱。没有审计日志的话，这段历史就永久丢失了。

### 论断3：重构的启示——四个必须

1. **必须去重**：飞书 message_id 唯一约束，拒绝飞书原始消息已被接收过
2. **必须校验**：任何 LLM 输出在写入数据库前都经过代码层校验（正/负、NaN、数值范围、不合理金额）
3. **必须审计**：所有操作（包括调账）写入 audit_log，保留完整追踪链
4. **必须拒绝**：Agent 必须学会说"我不理解这个"——错误输入宁可跳过不能编造

## 九、文章结构建议

### 标题方向

1. **《一条积分的进化之路——当一个 LLM Agent 在家庭里跑了两个月》**
2. **《从"学习积分 测试 +1 分"到"1. 口算+1分 2. 写字+3分"——一个 300 行 Agent 的驯化与被驯化》**
3. **《你的 Agent 有多"听话"？一个家庭积分系统的 50 天实录》**

### 核心叙事弧线

```
引子：一个 300 行的 OpenClaw handler，JSON 文件当数据库，就这么上线了
  │
  ▼
第1幕：盲测期（5天）
  用户像调 API 一样说话
  核心：格式试探、+0分综合症、重复输入逃生策略
  │
  ▼
第2幕：乱序期（11天）
  表达能力爆发但也溢出
  核心：4种消费句式、多项目列表、家庭方言盲区
  │
  ▼
第3幕：范式期（6天）
  双方找到了"对讲频道"
  核心：用户发明规则格式、"-"号歧义暴露、第一条调账
  │
  ▼
第4幕：秩序与崩坏期（16天）
  最日常也最危险
  核心：小数截断(静默错误)、×5重复记账(去重缺失)、测试攻击(防御缺失)
  │
  ▼
第5幕：默契与遗忘期（17天）
  信任重归，但深层机制失效
  核心：极简风格、[object Object]攻击、调账4次拉锯、复杂语义
  │
  ▼
尾声：重构的启示
  LLM 太"好用"了，反而成了风险
  Agent 治理的核心：代码层校验 > LLM 层理解
  余额可信吗？50天后的真相
```

### 关键论点（文章内核）

1. **AI 驯化是双向的**——表面上用户在使用 Agent，实际上 Agent 的行为模式也在重塑用户的表达方式。从"给王雯雯记一个积分吧"到"1. 口算+1分 2. 写字+3分"，用户的表达越来越"结构化"。

2. **LLM 作为核心引擎的脆弱性**——"test→扣10.5分""undefined→NaN"说明 LLM 代理最大的风险不是"做不了"，而是"太能做了"。没有边界校验的 LLM 代理，本质上是一个过度配合的实习生。

3. **Agent Runtime 治理=给 AI 加护栏**——原系统 300 行代码，无校验、无去重、无审计。新系统的关键词是：**代码层校验、SQLite 事务、audit_log、message_id 去重、两阶段调账**。治理不是限制 AI，而是让 AI 出错时可以追溯和修复。

4. **静默错误比显式错误更致命**——LLM 解析错了也没报错，数据被污染了没人知道。这是 LLM 作为"记分员"的根本缺陷：**它不报错，它直接出错。**
